## 1. 구축 목표

하루 수백 편씩 쏟아지는 arXiv 신규 논문 중 개인 관심사와 관련된 논문만 골라 추천하는 시스템의 **기반 데이터 인프라**를 구축한다. 구체적으로:

1. 대상 카테고리의 논문 메타데이터(제목·초록·저자·카테고리 등)를 **전수 수집**하고 매일 자동 갱신
2. 초록 임베딩 기반 **유사도 검색**이 가능한 벡터 인덱스 구축
3. 팀원 3인이 **하나의 공유 DB**를 실시간으로 사용할 수 있는 접근 체계 마련
4. 무료 스택만 사용 (arXiv API, SQLite, Chroma, sentence-transformers, FastAPI — 유일한 비용은 EC2 인스턴스)

## 2. 시스템 구성

### 2-1. 전체 구조

```
 arXiv API ◄──┐                    ┌─ 팀원 PC #1 (검색/수집)
              │   AWS EC2 인스턴스  │
        main.py (cron --daily)     ├─ 팀원 PC #2 (검색/수집)
        collector → preprocess     │
              ▼                    └─ 팀원 PC #3 (조회)
        SQLite (papers.db)              ▲
              ▼                         │ HTTP + X-API-Key
        embedder (bge-small)            │
              ▼                         │
        Chroma (벡터 인덱스) ◄── FastAPI(app.py, :8000)
```

### 2-2. 구성 요소별 역할

| 구성 요소 | 기술 | 역할 |
|---|---|---|
| 수집기 | Python `arxiv` 라이브러리 | 카테고리·기간 조건으로 메타데이터 조회 |
| 정형 저장소 | SQLite (`papers.db`) | 논문 메타데이터의 정본(source of truth) |
| 벡터 저장소 | Chroma (PersistentClient) | 초록 임베딩, 코사인 유사도 검색 |
| 임베딩 모델 | BAAI/bge-small-en-v1.5 (384차원) | 초록 → 벡터. 논문당 1회만 계산 |
| 공유 API | FastAPI + uvicorn | 검색/조회/수집/삭제의 유일한 창구, API Key 인증 |
| 자동화 | cron (`main.py --daily`) | 매일 신규분 증분 수집 |

**설계 원칙**: DB는 팀원 PC에 직접 노출하지 않고 모든 접근을 FastAPI로 일원화했다. 전처리·버전 비교 규칙이 서버 한 곳에만 존재해 클라이언트-서버 로직 불일치를 원천 차단하고, 팀원 PC에는 무거운 ML 의존성(torch 등)을 설치할 필요가 없다.

## 3. 수집 설계

### 3-1. 수집 대상

| 시점 | 카테고리 | 내용 |
|---|---|---|
| 초기 구축 | cs.RO, cs.CV, cs.CL | 팀 관심사(로봇·비전·NLP) 기준 |
| 개편 검토 | cs.LG, cs.IR 후보 실측 | cs.LG(ML)는 하루 ~90건으로 과대, cs.IR은 하루 ~10건으로 고밀도 벤치마크 후보로 검토 |
| **최종 (현재)** | **cs.RO, cs.CV, cs.CL 유지** | 팀 논의 결과 초기 구성 유지로 확정. 기존 수집 데이터를 그대로 활용 가능(데이터 정리 불필요)하고, cs.CL 프로필 4개(직업군 기반)로 재구성 |

카테고리별 최근 7일 제출량 실측(2026-07-18, 크로스리스트 포함): cs.LG 625건, cs.CV 501건, cs.CL(참고) ~500건 추정, cs.RO 242건, cs.IR 72건.

### 3-2. 수집 경로 3가지

| 경로 | 스크립트 | 용도 |
|---|---|---|
| 초기 백필 | `main.py --backfill N` (서버) | 최근 N개월치 일괄 수집 |
| 일일 증분 | `main.py --daily` (서버, cron) | 마지막 실행 이후 신규분만 |
| 분산 수집 | `collect_and_push.py` / `colab_client.ipynb` (팀원) | 특정 기간·카테고리를 팀원 PC/코랩에서 수집해 서버로 push. 서버가 정제·임베딩 수행 |

### 3-3. arXiv API 제약과 대응

| 제약 | 내용 | 대응 |
|---|---|---|
| 페이지네이션 한계 | 한 쿼리 결과가 약 10,000건을 넘으면 HTTP 500 | 기간을 **7일 단위 창**으로 분할 조회 (`fetch_by_date_range_chunked`) |
| Rate limit / 일시 장애 | 장시간 수집 중 429/503 발생 | 창 단위 **지수 백오프 재시도**(최대 5회, 30초×시도 횟수). 최종 실패한 창은 건너뛰고 재수집 명령 안내 |
| 호출 예절 | 요청 간 3초 이상 대기 권장 | `delay_seconds=3.0`, 페이지 크기 100 고정 |

## 4. 전처리·저장 설계

### 4-1. 전처리 (`preprocess.py`)

1. **중복 제거**: 같은 논문이 카테고리별 조회에 중복 등장 → arxiv_id 기준으로 병합, **최고 버전만 유지**, 카테고리는 합집합
2. **초록 클린업**: LaTeX 수식(`$...$`)·명령어(`\textbf{}` 등)·줄바꿈·중복 공백 제거 → 임베딩 입력용 `abstract_clean` 생성 (원문은 `abstract_raw`로 별도 보존)

### 4-2. SQLite 스키마 핵심

- PK = **버전 제거된 arxiv_id** (예: `2401.12345`) → 버전 중복 자동 방지
- `version` 컬럼으로 버전 비교 upsert: 더 높은 버전이 오면 내용 갱신 + `embedded=0` 리셋(재임베딩 큐 등록)
- `embedded` 플래그(0/1)가 임베딩 대기열 역할 — 논문당 1회만 계산 보장
- 인덱스: `submitted_date`, `primary_category`, `embedded`

### 4-3. Chroma 벡터 인덱스

- id를 SQLite와 동일한 arxiv_id로 사용 → 검색 결과를 SQLite 메타데이터와 join
- 코사인 거리(`hnsw:space: cosine`), 벡터는 정규화 후 저장
- metadata에 `{title, primary_category, submitted_date}` 최소 복제 → SQLite 조회 없이 카테고리 필터 검색 가능
- **임베딩 모델별 컬렉션 분리**: 모델을 바꿔 실험해도 기존 벡터 보존 (모델 비교실험 지원)

## 5. 공유 API

| 엔드포인트 | 기능 | 인증 |
|---|---|---|
| `GET /health` | 생존 확인 + 논문 수 | 불필요 |
| `POST /search` | 텍스트 쿼리 임베딩 → 유사 논문 top-k (카테고리 필터 지원) | X-API-Key |
| `POST /papers/ingest` | 팀원 수집분 반영 (서버가 정제·upsert·임베딩) | X-API-Key |
| `GET /papers/{id}`, `GET /papers?ids=` | 상세 조회 (단건/다건) | X-API-Key |
| `POST /papers/delete` | id 목록 또는 primary_category 기준 삭제 (SQLite+Chroma 동시). `confirm=true` 필수, 전체 삭제는 서버 CLI(`reset_data.py`) 전용 | X-API-Key |

클라이언트: `client_example.py`(로컬), `colab_client.ipynb`(코랩 — 검색/후보 CSV 추출/수집/삭제, API 키는 실행 시 입력받아 파일에 미저장).

## 6. 구축 결과 (현황)

### 6-1. 데이터 규모

| 항목 | 수치 | 기준 |
|---|---|---|
| 서버 보유 논문 | 약 60,000건 | 팀 공유 DB (EC2) |
| 로컬 사본 | 19,991건 | 2026-04-07 ~ 07-02 수집분 |
| 로컬 임베딩 완료 | 11,712건 | bge-small, 384차원 |

### 6-2. 로컬 사본 카테고리 분포 (primary 기준 상위)

| 카테고리 | 건수 | 비고 |
|---|---|---|
| cs.CV | 8,442 | 수집 대상 |
| cs.CL | 5,076 | 구 수집 대상 (개편으로 제외) |
| cs.RO | 2,890 | 수집 대상 |
| cs.LG | 1,235 | 크로스리스트 유입 |
| cs.AI 외 70여 종 | ~2,350 | 크로스리스트 유입 |

> **크로스리스트 유입**: arXiv 카테고리 쿼리는 해당 카테고리에 교차 등록된 논문도 반환하므로, 수집 대상이 3개여도 primary 카테고리는 75종에 달한다. 이는 오류가 아니라 arXiv 구조의 자연스러운 결과다.

### 6-3. 검색 품질 검증 (표본)

P1 프로필(언어 조건 로봇 매니퓰레이션)로 top 30을 추출해 초록 전수 판정한 결과:
- **관련(2점 이상) 27/30 = 90%**, 무관·제외조건 위반 0건
- 프로필의 세부 관심사("모호한 지시 해석")에 대응하는 논문들이 상위권에 정상 랭크
- 주변부 3건(인프라·RL 보상 설계 등)은 임베딩 검색의 알려진 한계로, B단계 재랭커의 학습 목표가 됨

## 7. 데이터 정합성 관리

| 위험 | 관리 방법 |
|---|---|
| SQLite-Chroma 불일치 | 삭제 시 양쪽 동시 처리(`delete_vectors`가 모든 컬렉션 순회). 플래그 불일치 발견 시 Chroma id 기준으로 SQLite 플래그 복구 (실제 1회 수행: 플래그 0건 → 11,712건 복구) |
| 동시 쓰기 경합 | SQLite 연결 `timeout=30`으로 대기 허용 |
| 실수 삭제 | API는 `confirm=true` + id/카테고리 단위만 허용, 전체 삭제는 서버 CLI에서 'delete' 타이핑 확인 필수 |
| 비밀값 유출 | API 키는 환경변수/`local_config.py`(.gitignore)/코랩 입력창으로만 취급, 커밋 파일에 미포함 |

## 8. 한계 및 후속 작업

**한계**
1. 임베딩 유사도만으로는 복합 조건(포함+제외) 처리 불가 → B단계 재랭커로 해결 예정
2. 키워드(정확 용어) 검색 미지원 → SQLite FTS5 기반 하이브리드 검색 추가 예정 (설계 완료, [pipeline.md](pipeline.md) 참고)
3. 초록만 수집 (본문·인용 관계 없음) — 인용 데이터가 필요해지면 Semantic Scholar API 연동 검토

**즉시 후속 작업**
1. 남은 미임베딩분(로컬 8,279건) 일괄 임베딩
2. 프로필 11개(cs.RO 3 / cs.CV 4 / cs.CL 4)의 키워드 히트 수 검증 (50~500 구간)
3. 시드 논문 수집(팀원별 10~20편) — 평가 hold-out과 콜드스타트용

> 참고: 카테고리를 유지하기로 하면서 데이터 정리·재백필은 불필요해졌다. 개편 검토 과정에서 만든 삭제 도구(`reset_data.py`, `POST /papers/delete`)는 향후 카테고리 변경이나 오염 데이터 정리 시 사용할 수 있는 자산으로 남는다.
