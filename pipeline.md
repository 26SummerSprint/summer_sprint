## 1. 현재 상태와 목표

| 항목 | 내용 |
|---|---|
| 완료된 것 (A단계) | arXiv 수집 파이프라인, SQLite + Chroma 저장, FastAPI 공유 서버, 약 6만 건 논문 데이터 |
| 현재 검색 방식 | `embedder.search_similar()` — zero-shot 임베딩 코사인 유사도 검색만 지원 |
| 목표 | 하루 수백 편 신규 논문 → 관심사 기반 top 5~10편 선정 + 추천 이유 한 줄 생성 |
| 프로젝트 요구 | 실동작하는 도구 완성 + **모델 선택을 정당화하는 비교실험** 수행 |
| 제약 | 3인 팀, 5~6주, **사람 라벨링 여력 거의 없음** |

### 현재 방식(zero-shot 임베딩)의 한계

| 한계 | 설명 | 예시 |
|---|---|---|
| 복합 조건 처리 불가 | 일반 문장 유사도로 학습된 bge 계열은 "포함 + 제외" 복합 조건에 약함 | "VLM 로봇 매니퓰레이션 관심, **순수 RL 이론은 제외**" → 제외 조건이 반영 안 됨 |
| 부정문 이해 부족 | 임베딩 공간에서 부정문과 긍정문이 가깝게 위치 | "X가 아닌 것" 검색 시 X가 상위에 나옴 |
| 추천 이유 없음 | 유사도 점수만 반환 | "왜 이 논문이 나에게 중요한가" 설명 불가 |

## 2. 파이프라인 전체 설계

### 2-1. 4단계 구조

```
[관심사 프로필 (자연어)]
        │ 임베딩
        ▼
┌──────────────────────────────────────────────────┐
│ Stage 1. Retrieval          (기존 코드 재사용)      │
│  오늘자 신규 논문 전체 vs 프로필 벡터 cosine sim     │
│  → 후보 50~100편                                   │
└──────────────────┬───────────────────────────────┘
                   ▼
┌──────────────────────────────────────────────────┐
│ Stage 2. Rerank             (★ 신규 학습 대상)     │
│  (프로필, 초록) pair → relevance score             │
│  → 후보 20~30편으로 압축                            │
└──────────────────┬───────────────────────────────┘
                   ▼
┌──────────────────────────────────────────────────┐
│ Stage 3. LLM 최종 선정 + 이유 생성  (프롬프팅)       │
│  후보 20~30편 + 프로필 → top 5~10 + 추천 이유 한 줄 │
└──────────────────┬───────────────────────────────┘
                   ▼
┌──────────────────────────────────────────────────┐
│ Stage 4. 피드백 루프         (확장 단계)            │
│  👍/👎 → 재랭커 재학습 라벨 + 프로필 벡터 업데이트    │
└──────────────────────────────────────────────────┘
```

### 2-2. 단계별 상세

| Stage | 역할 | 모델/방법 | 학습 여부 | 입출력 |
|---|---|---|---|---|
| 1. Retrieval | 대량 후보 추출 (recall 확보) | 임베딩 모델 (bge-small 등) + Chroma | ❌ (비교실험만) | 신규 논문 수백 편 → 50~100편 |
| 2. Rerank | 정밀 재정렬 (제외 조건 등 반영) | 경량 cross-encoder 또는 임베딩 + scoring head | ✅ **핵심 학습 대상** | 50~100편 → 20~30편 |
| 3. LLM 선정·설명 | 최종 선정 + 추천 이유 생성 | LLM 프롬프팅 (listwise vs pointwise 비교) | ❌ (프롬프트 실험) | 20~30편 → 5~10편 + 이유 |
| 4. 피드백 | 개인화 지속 개선 | 라벨 축적 + 프로필 벡터 EMA 업데이트 | ✅ (재학습) | 👍/👎 → 학습 신호 |

### 2-3. Stage 2(재랭커)가 필요한 이유

- Stage 1의 임베딩 검색만으로는 "이 주제는 포함, 이 하위분야는 제외" 같은 복합 조건을 걸러낼 수 없음
- 벤치마크 라벨(또는 LLM pseudo-label)로 fine-tune한 재랭커가 이 격차를 메움
- Stage 3에 넘기는 후보 수를 줄여 **LLM 호출 비용도 절감**

### 2-4. Stage 3 프롬프트 전략 비교 대상

| 전략 | 방식 | 장점 | 단점 |
|---|---|---|---|
| Listwise | 후보 전체를 한 프롬프트에 넣고 순위 매김 | 호출 1회, 후보 간 상대비교 가능 | 컨텍스트 길이 부담, 위치 편향 |
| Pointwise | 논문 1편씩 개별 스코어링 | 안정적, 병렬 가능 | 호출 수 = 후보 수 (비용↑) |

## 3. 손실함수 / 학습함수 / Objective Function

학습이 실제로 들어가는 지점은 **Stage 2 재랭커**. 데이터는 `(프로필, 논문, 라벨)` 형태이며, 라벨 양이 매우 제한적이라는 것이 설계의 핵심 제약.

### 3-1. 제안 비교

| # | 제안 | Objective | 라벨 요구량 | 적합 상황 |
|---|---|---|---|---|
| 1 | **Contrastive / InfoNCE** (기본 추천) | `L = -log( exp(sim(p,d⁺)/τ) / Σᵢ exp(sim(p,dᵢ)/τ) )` | 적음 (in-batch negative 활용) | positive만 확보돼도 시작 가능. bi-encoder를 fine-tune하면 Stage 1 recall도 동반 개선 |
| 2 | **Pairwise (RankNet)** | `L = log(1 + exp(-(sᵢ - sⱼ)))` | 중간 (쌍대비교 라벨) | "관련/무관" 절대 기준 합의가 어려울 때 — 상대비교가 라벨러 간 더 일관적 |
| 3 | **Listwise (ListNet / LambdaRank)** | NDCG@10에 직접 최적화 | 많음 (리스트 단위 라벨) | 배포 목표("상위 5~10 우선순위")와 학습 목표가 일치. 데이터 규모 확인 후 채택 |
| 4 | **LLM 증류** (실질적으로 가장 중요) | `L = L_hard(사람 라벨) + λ·L_soft(LLM teacher 점수)` | 사람 라벨 최소화 | **라벨 부족을 LLM pseudo-label로 보완.** 60k 논문 + LLM 자원을 가장 잘 활용 |

### 3-2. 제안 4 (LLM 증류) 세부 구성

| 구성 요소 | 내용 |
|---|---|
| Teacher | 강한 LLM이 (프로필, 논문) 쌍에 관련성 점수 부여 → 수천~수만 쌍 pseudo-label |
| Student | 경량 재랭커 (base 임베딩 모델 + 얇은 head) |
| 사람 라벨 역할 | teacher 품질 검증(agreement 체크) + 최종 캘리브레이션 (수백 쌍이면 충분) |
| Hard negative | 주제는 겹치지만 프로필이 명시적으로 제외한 논문 — **라벨링 가이드에 "제외조건 위반 예시"를 의도적으로 포함**해야 확보됨 |

### 3-3. 학습 절차 (하이퍼파라미터)

| 항목 | 권장값 | 비고 |
|---|---|---|
| Base 모델 | 현재 임베딩 모델(bge-small) 또는 비교실험 승자 | 위에 선형/2-layer MLP head |
| Optimizer | AdamW | — |
| Learning rate | 2e-5 ~ 1e-4 | head만 학습 시 큰 쪽, 전체 fine-tune 시 작은 쪽 |
| Warmup | 전체 스텝의 5~10% | — |
| Epoch | 3~5 | 데이터가 작아 과적합 위험 → early stopping 병행 |

### 3-4. 데이터 분리 원칙 (중요)

| Split 방식 | 목적 | 비고 |
|---|---|---|
| **시간 기준 분리 (필수)** | 논문 제출일 기준으로 train/test 분리 | "미래 논문을 예측"하는 실제 배포 상황 재현. **랜덤 split 금지** |
| 프로필 기준 분리 (권장) | 학습에 없던 새 관심사에 대한 일반화 확인 | 팀원 프로필 중 1개를 hold-out |

## 4. 평가 설계 (사람 라벨링 최소화)

**전략**: 라벨을 없애는 게 아니라, ① 라벨을 싸게 만들고(LLM-as-judge) ② 라벨 없이 되는 프록시 평가를 조합.

### 4-1. LLM-as-Judge — 메인 평가 (가장 추천)

| 항목 | 내용 |
|---|---|
| 방법 | 강한 LLM에게 (프로필, 초록) 쌍 → 관련성 0~3점 부여 → **평가용 골드 라벨 자동 생성** |
| 규모/비용 | 프로필 3개 × 후보 200편 = 600회 호출. 초록만 입력 시 호출당 수백 토큰 → 몇 달러 수준 |
| **검증 절차 (필수)** | 팀원 3명이 각자 자기 프로필에 대해 30~50편만 직접 라벨링 → judge와 agreement(Cohen's κ) 측정. 1인당 1~2시간 |
| 방어 논리 | "judge가 사람 라벨과 N% 일치함을 확인 후 나머지 평가를 대행시켰다" — 보고서에서 방법론적으로 방어 가능 |
| ⚠️ 주의 | Stage 3 선정 LLM과 judge LLM은 **다른 모델** 사용 (같으면 self-preference bias로 평가 오염) |

### 4-2. 라벨 불필요 프록시 평가 — 보조

| 방법 | 절차 | 지표 |
|---|---|---|
| **시간 기반 자기지도 평가** | ① 각자 실제로 읽은/북마크한 논문 10~20편 시드 등록 → ② 절반으로 프로필 구성 → ③ 나머지 절반이 60k 코퍼스에서 상위 랭크되는지 측정 | Held-out hit rate, MRR |
| **분포 sanity check** | 추천 top 10의 카테고리/키워드 분포가 프로필과 정합한지 확인 | 분포 정합도 |
| **제외조건 위반율** | "순수 RL 이론 제외" 등 위반 논문 비율을 키워드 매칭으로 근사 측정 | 위반율 (재랭커 학습 효과를 보여주기 좋음) |

> 시드 논문 수집은 라벨링이 아니라 "내가 아는 논문 목록 적기"라 부담이 거의 없고, 진짜 관심사를 반영한 positive임.

### 4-3. 단계별 평가지표 정리

| Stage | 지표 | 골드 라벨 출처 |
|---|---|---|
| Stage 1 (Retrieval) | Recall@50 | LLM judge / held-out 시드 논문 |
| Stage 2 (Rerank) | NDCG@10, Precision@10 | LLM judge |
| Stage 3 (LLM 선정) | Precision@10 + 최소 1회 사람 평가("실제로 유용했나") | LLM judge + 사람 |
| 설명 품질 | 충실성/관련성 (선택) | 사람 소수 평가 |

### 4-4. 사람 라벨의 역할 재정의

| 용도 | 기존 계획 | 재정의 후 |
|---|---|---|
| 학습 데이터 | 사람 라벨 수천 건 | LLM teacher pseudo-label (§3-1 제안 4) |
| 평가 골드셋 | 사람 라벨 수백 건 | LLM judge 라벨 600건 |
| 사람의 역할 | 대량 라벨링 | judge 검증 30~50건/인 + 블라인드 비교 10건/인 |
| **총 작업량** | 1인당 수일 | **1인당 2~3시간** |

## 5. 모델 비교실험 설계

"모델 선택 정당화" 요구를 충족하는 실험. 절대 점수보다 **동일 쿼리 상대평가(A/B)**가 저렴하고 충분함.

### 5-1. Stage 1 — 임베딩 모델 비교

| 후보 | 특징 |
|---|---|
| BAAI/bge-small-en-v1.5 (현재) | 빠름, 384차원. 현재 기본값 |
| BAAI/bge-base-en-v1.5 | 품질↑ 속도↓ |
| intfloat/e5 계열 | 범용 검색 강세 |
| **SPECTER2** | **논문 임베딩 전용 학습 모델 — 강력한 베이스라인 후보** |

**비교 방법**: 같은 프로필로 각각 top 20 추출 → ① LLM judge 쌍대비교("어느 리스트가 더 적합한가") ② held-out hit rate 비교. 사람은 최종 승자 확인용 블라인드 비교만 (1인당 10케이스).

> 인프라 참고: `config.py`가 임베딩 모델별로 Chroma 컬렉션을 자동 분리하므로, 모델 교체 실험 시 기존 벡터가 보존됨.

### 5-2. Stage 3 — LLM 비교

| 축 | 비교 대상 |
|---|---|
| 모델 크기 | 대형 vs 중형 vs 소형 — 비용·품질 트레이드오프 |
| 프롬프트 전략 | listwise vs pointwise (§2-4) |
| 지표 | Precision@10, 추천 이유 품질, 호출 비용, 지연시간 |

## 6. Task 확장 제안

### 6-1. 우선순위 높음 (5~6주 내 현실적)

| Task | 내용 | 연계 |
|---|---|---|
| 피드백 루프 | 👍/👎 → 재랭커 재학습 라벨 축적 + 프로필 벡터 EMA 업데이트 | Stage 4 |
| 다양성 재랭킹 | top 5~10이 한 서브토픽에 몰리지 않게 MMR 적용 | Stage 2~3 사이 |
| 콜드스타트 | 신규 유저: 관심사 텍스트 키워드 추출 or "예시 논문 몇 편 등록"으로 부트스트랩 | §4-2 시드 방식 재활용 |
| LLM 비교실험 확장 | 크기/프롬프트별 비용·품질 트레이드오프 | §5-2, 모델 선택 정당화와 직결 |

### 6-2. 선택 (시간 남으면)

| Task | 내용 |
|---|---|
| 멀티 관심사 프로필 | facet별 클러스터링 후 각각 추천 |
| 설명 품질 벤치마크 | 추천 이유의 충실성/관련성 사람 평가 |
| 관심사 드리프트 대응 | 주기적 프로필 재확인 알림 |
| 배포 채널 확장 | Slack/이메일 데일리 다이제스트, 카테고리 확장 |

## 7. 실행 우선순위 요약

| 순서 | 할 일 | 산출물 |
|---|---|---|
| 1 | **LLM judge 구축** + 팀원별 30~50편 검증 라벨링 | 평가 골드셋 600건 + agreement 수치 |
